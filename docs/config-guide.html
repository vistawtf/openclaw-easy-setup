These files are already configured — don't start from scratch

What makes the starter pack different from OpenClaw's defaults

OpenClaw ships with minimal defaults — no personality, no memory routing, no quality standards. Users who start from scratch typically spend weeks figuring out what to add and why.

The starter pack skips that. When you run the setup wizard, BOOTSTRAP.md walks your assistant through a short onboarding conversation that personalizes the key files for you before you even open them. What you get after that:

  • SOUL.md — a real communication style: no sycophantic openers, no padding, no filler. Includes anti-patterns to avoid (em dashes, significance inflation, rule-of-three lists) and model escalation rules so it uses the right model for each task.
  • AGENTS.md — 15 operational rules covering memory routing, execution routing, quality gates, cron reliability, handoff protocol, and more. Things you'd otherwise discover by hitting the failure each rule prevents.
  • OPS.md, HEARTBEAT.md, SECURITY.md — sensible defaults for the operational patterns most people need eventually.

This guide is about understanding what's already there and adjusting the parts specific to you — your name, timezone, projects, preferred tone. The universal behaviors are already handled.

IDENTITY.md

Give your AI a name and personality base
Required

What this file does

This defines your AI's name and base persona. The name matters more than it sounds: you'll use it constantly, and it gives coherence to every interaction.

What to fill in

  • Name — Pick something you'd naturally say: "Hey Nova, ..." or "Atlas, can you check...". Examples: Jarvis, Atlas, Nova, Max, Sage, Echo.
  • Vibe — One sentence describing the character. Examples: "Direct and practical, with dry humor" / "Calm and thorough, like a senior engineer"
  • Emoji (optional) — A visual shorthand. 🤖 ⚡ 🧠 🦉
  • Avatar (optional) — A URL or local path to an image, used in some UIs.
Tip Choose a name you'll use consistently. Some users name it after a character they like (Jarvis, HAL, Friday). Others pick something neutral (Sage, Pilot). Either works — just pick one.

USER.md

Tell your AI who you are
Required

What this file does

Without this file, your AI asks you the same questions every session. With it, it already knows your context from the first message. Think of it as a one-time briefing that stays loaded.

What to fill in

  • Name — How you want to be addressed.
  • Timezone — Needed for crons and scheduling. Use a format like UTC-4 or America/New_York.
  • Work Hours — Helps the AI avoid alerting you at 2am.
  • Language — Your preferred language. If you have a regional dialect (Mexican Spanish, Brazilian Portuguese), specify it — AI models default to generic forms otherwise.
  • What I Do — A short paragraph about your work context. This makes the AI's answers much more relevant.
  • Current Priorities — What you're focused on right now. The AI uses this to prioritize when tasks conflict.
  • Active Projects — List them with a one-liner each.
  • Preferences — How you like to work. "Give critical feedback", "keep things organized", "don't over-explain".

What not to put here

Don't put API keys, passwords, or sensitive credentials here. This file is loaded into the AI's context, so anything here is visible in the conversation. Keys belong in ~/.config/.

Tip You don't need to fill everything on day 1. Start with Name, Timezone, Language, and What I Do. Add more as you discover what context makes the AI more useful for you.

Common mistake

Timezone matters more than you think If you skip timezone, scheduled tasks (crons) will run at the wrong time, often UTC instead of your local time. Set it explicitly.

SOUL.md

The most important file — define how your AI talks and thinks
Required

What this file does

This is the personality layer. It controls how your AI communicates: tone, feedback style, language, and which writing patterns to avoid.

What to customize

How you talk — add your style preferences after the existing rules. Examples:

"Be concise but warm — like texting a smart friend"
"Use analogies from sports to explain complex ideas"
"Challenge my assumptions when you see weak logic"
"Default to bullet points over paragraphs"

Opinions — decide how direct you want the AI to be:

"Sugar-coat nothing — treat me like a peer"
"Be diplomatic but honest — I'm sensitive to bluntness"
"Challenge me on strategy, be gentle on personal stuff"

Language — set your language rules clearly:

# English only:
"Always respond in English."

# Bilingual:
"Match whatever language I use."

# Regional Spanish (important!):
"Use Argentine voseo: tenés, querés, podés."
"Use Mexican tuteo. Occasional 'güey' is fine."
"Use Colombian tuteo with usted for formal contexts."

Vibe — the overall character in one line:

"Like a smart coworker who respects my time"
"The engineer friend who always has a better way"
"Calm advisor, not an eager intern"

What not to touch

The Linguistic patterns to avoid section — keep it exactly as-is. These rules suppress the most common AI writing artifacts (em dashes, rule-of-three, significance inflation, etc.). They work universally and improve every response.

The Communication Rules section (No Leaked Reasoning, No Double Messages) — these are operational safeguards, not style preferences. Keep them.

SOUL.md is a living document Whenever the AI's behavior bugs you, the fix is here. Add a rule, save the file, done. You'll refine it over time — everyone does.
If you use a regional language Be specific. "Spanish" defaults to generic Castilian. If you're Argentine, Mexican, Colombian, or from another region, say so. The difference between tuteo and voseo in a conversation is jarring.

AGENTS.md

Execution rules — what your AI does and how
Required

What this file does

This is the operating system of your assistant. While SOUL.md defines personality, AGENTS.md defines behavior. It tells the AI how to handle tasks, where to save things, when to act independently vs. ask you, and what the quality bar is before saying "done".

What to review and customize

Section 2, Memory Routing: the table defines where different types of information get saved. The defaults work well. You can add rows as you add projects. Example:

| Project notes | myproject/CONTEXT.md |
| Code patterns | memory/solutions/code/ |

Section 3, Task Management: if you use Notion, fill in your database ID and uncomment the section. If you don't use Notion, delete it — simpler is better.

Section 4, The 2-3 Minute Rule: this tells the AI when to spawn a subagent vs. work directly. The defaults are sensible. Adjust based on how much you care about main session availability.

What not to touch

  • Section 1, Session Startup: don't modify this. It's the boot sequence.
  • Section 5, Quality Gate: keep these rules — they prevent false "done" reports.
  • Section 6, Date/Time Verification: keep it. AI models hallucinate dates without this.
  • Section 7, Safety Baseline: keep it. These are guardrails.
  • Section 8, Communication Rules: keep it. These prevent annoying behaviors.
Start minimal Don't add rules you don't need yet. AGENTS.md grows over time as you discover what behaviors you want. The defaults handle 80% of cases for most users.

TOOLS.md

Document your APIs, services, and setup notes
Optional at first

What this file does

Your AI reads this every session. Anything you document here, it knows automatically — no need to re-explain your setup every conversation. Think of it as your personal cheat sheet for infrastructure: API key locations, channel IDs, device names, useful patterns.

API key storage

Store credentials in ~/.config/<service>/, not in this file. TOOLS.md only documents where keys live and how to use them.

mkdir -p ~/.config/notion
echo "YOUR_API_KEY" > ~/.config/notion/api_key
chmod 600 ~/.config/notion/api_key

What to add here

  • Your Telegram chat_id (needed for crons to message you)
  • Notion database ID and property names
  • SSH host aliases
  • Any "never forget this about my setup" notes
  • API services you've connected and their status

When to fill it

Fill it progressively. When you connect a new service, document it here. When you discover a useful pattern or a "never do X" lesson, add it. After a month it accumulates enough context to save you significant setup time each session.

Never put API keys directly in this file TOOLS.md is loaded into AI context. A key in this file could be exposed in conversation logs. Always store keys in ~/.config/ and only reference their location here.

OPS.md

Operational playbook — crons, quality, workflows
Optional

What this file does

OPS.md is the detailed operational reference. AGENTS.md has the rules that always apply. OPS.md has the how-to details that only matter in specific situations. Think of AGENTS.md as the constitution and OPS.md as the implementation guide.

What's already there

  • Cron Golden Defaults — the exact flags required for crons to actually work. This section is pre-configured with patterns that prevent the most common cron failures (silent failures, wrong timing).
  • Quality Gate — checklist the AI runs before saying any task is done.
  • Subagent Spawn Rules — when to spawn vs. work directly.
  • Handoff Protocol — how completed work gets documented in daily logs.

When to touch it

You don't need to customize OPS.md on day 1. It's most useful when:

  • You set up your first cron — read the Cron Golden Defaults section carefully
  • You have a repeating workflow you want to standardize
  • You're building with code and want repo hygiene rules
Cron failures are silent by default Without --wake now and --announce, crons may run but you'll never see output. The defaults in OPS.md prevent this. Read that section before creating your first scheduled task.

HEARTBEAT.md

Instructions for periodic background checks
Optional

What this file does

The heartbeat is a periodic check your AI does on its own, independent of your messages. Every X minutes, it wakes up, reads this file, and decides whether to alert you or stay quiet. It's what makes your AI proactive instead of purely reactive.

Without vs. with heartbeat

Without heartbeat With heartbeat
AI only responds when you message it AI proactively surfaces things that need attention
You have to manually check everything AI alerts you to failing services, calendar events, etc.
Memory maintenance is manual AI organizes memory files during quiet heartbeats

What to customize

The "When to Alert Me" and "When to Stay Quiet" sections are sensible defaults. The part you'll want to configure is "What to Check" — add the specific things you want your AI to monitor:

- Email — any urgent unread messages?
- Calendar — upcoming events in next 24-48h?
- GitHub — any failing CI or PR reviews waiting?
- Services — is my app responding to health checks?

When to set this up

Skip it for week 1. Once you're comfortable with the basics, set up a heartbeat cron via:

openclaw cron create \
  --name "Heartbeat" \
  --cron "*/15 * * * *" \
  --message "Run heartbeat check per HEARTBEAT.md" \
  --to telegram:YOUR_CHAT_ID \
  --announce \
  --wake now

Security — SECURITY.md

Run through this before publishing or sharing your workspace
Before sharing

Why this matters

OpenClaw gives your AI broad access to your system: files, shell, network. Most users configure this for personal use on a trusted machine, and the risk is low. But if you ever share your workspace, publish it to git, or run OpenClaw on a server others can access, the attack surface changes significantly.

The most important rules

  • No API keys in any workspace file — store them in ~/.config/<service>/ with chmod 600
  • Run openclaw secrets audit --check — clean before sharing
  • Create a .gitignore before your first commit (template is in SECURITY.md)
  • Run the hardening script (in SECURITY.md) after initial setup
  • Review openclaw security audit output — fix any critical findings

Quick secret scan

Run this before committing or sharing anything:

grep -rn "sk-ant\|sk-or\|sk-proj\|ghp_\|gsk_\|xoxb-\|xoxp-" ~/.openclaw/workspace/

If this returns anything, you have a key in the wrong place.

If a key is ever exposed

Revoke first, clean later If a key ends up in a public git repo, revocation is the only thing that matters. Cleaning the git history without revoking does nothing — bots scrape GitHub for keys within minutes of a push. See the Token Revocation Quick Reference table in SECURITY.md.

Tool access restriction

By default, the AI has broad access. You can restrict it in ~/.openclaw/openclaw.json:

{
  tools: {
    fs: { workspaceOnly: true },      // limit file access to workspace only
    exec: { security: "allowlist" },  // restrict shell commands
    elevated: { enabled: false },     // no host-level execution
  }
}

For teams

OpenClaw is designed as a personal assistant tool with one trust boundary per gateway. If multiple people need access:

  • Run separate gateways per person (recommended)
  • Or use a dedicated machine with sandboxing enabled and no personal accounts signed in
  • Never share a gateway where one person can see another's conversation history
Key rotation schedule Model API keys (Anthropic, OpenAI): every 90 days. Bot tokens (Telegram, Discord): every 180 days. Integration tokens (Notion, etc.): every 180 days. See the full table in SECURITY.md.

memory/MEMORY.md

Long-term memory — starts empty, fills itself
Auto-managed

What this file does

This is where your AI stores long-term learnings: decisions you made, lessons from mistakes, preferences it discovered. You don't manage it manually — the AI writes to it automatically when it learns something worth keeping.

What goes here vs. daily logs

memory/MEMORY.md memory/YYYY-MM-DD.md
Curated long-term knowledge Raw daily events and logs
Strategic decisions and rationale What happened today
Lessons learned from past mistakes Task completions and notes
Preferences discovered over time Context from today's sessions

Maintenance

Every 2-3 weeks, open MEMORY.md and do a quick curation pass:

  • Delete entries that no longer apply
  • Consolidate repeated patterns into one rule
  • Keep it under ~150 lines — beyond that it's noise, not signal
You don't need to touch this file on day 1 MEMORY.md ships empty. Your assistant fills it automatically as you work together. You don't need to write anything there manually. Just know it exists and review it occasionally.

After setup — what's next

How your setup grows over time

Once your config files are set, a few things worth doing when you're ready:

  • Connect services to TOOLS.md as you need them (Brave Search, Notion, GitHub)
  • Set up a cron or two — a daily briefing, a weekly review
  • Install skills from clawhub.com as new use cases come up — read the SKILL.md before installing, community skills run with the same access as your assistant
  • Build custom skills for workflows specific to your domain
  • Run openclaw security audit and review SECURITY.md — worth doing early, not after something goes wrong
  • Curate memory/MEMORY.md periodically — the daily logs are raw notes, MEMORY.md is where the useful patterns live

The config files improve over time. Add rules to AGENTS.md when you notice a pattern. Adjust SOUL.md when the tone isn't right. The more specific they get, the more useful the assistant becomes.
